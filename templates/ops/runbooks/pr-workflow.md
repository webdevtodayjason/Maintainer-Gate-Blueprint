# Runbook: Pull Request Workflow

> Every PR must pass quality gates and notify the {{OPERATOR_ROLE}} before merge.

## PR Creation Checklist

Before creating any PR:

1. **Branch**: Work on a feature/fix branch (feat/, fix/, auto/bugfix-)
2. **Build**: `{{CI_BUILD_COMMAND}}` passes locally
3. **Changelog**: Updated if the change is significant
4. **Tests**: Run relevant tests
5. **Create PR**: `gh pr create` with detailed description

## Quality Gates (IN ORDER)

```
Create branch → Local build → Create PR → Gate 1: CI → Gate 2: Code Review → Gate 3: PR Email → Gate 4: Merge
```

### Gate 1: CI

After creating the PR, check CI status:
```bash
gh pr checks <pr-number> --repo {{GH_ORG}}/{{REPO_NAME}} --watch
```
If CI fails: read the error, fix it, push to the branch, wait for CI to re-run. Do NOT notify the {{OPERATOR_ROLE}} about a broken PR.

### Gate 2: Code Review

After CI passes, check for review comments:
```bash
gh pr view <pr-number> --repo {{GH_ORG}}/{{REPO_NAME}} --json reviews,comments
```
If the reviewer has suggestions: fix them, push, wait for re-review. Repeat until approved or no blocking comments.

### Gate 3: PR Email (ONLY after gates 1+2 pass)

Only send the PR notification email AFTER:
- CI is green (all checks pass)
- Code review has no blocking issues

```bash
bash scripts/send-pr-email.sh <pr-number>
```

**NEVER send the email on a red build or unresolved review comments.**

### Gate 4: Merge

- Bug fix PRs can auto-merge after all gates pass
- Feature PRs wait for {{OPERATOR_ROLE}} review

## PR Email Notification (MANDATORY)

After creating a PR and passing gates 1+2, send an email notification:

**To:** {{NOTIFICATION_EMAIL}}
**Subject:** `{{PROJECT_NAME}} PR #<number> — v<version>: <PR title>`

**Body includes (auto-generated by script):**
- PR title and number with link
- Branch name, author, additions/deletions summary
- PR description
- Changed files diff stat
- Changelog section (if applicable)

### How to Send

```bash
bash scripts/send-pr-email.sh <pr-number>
```

## Who Must Follow This

**Everyone.** Human developers, AI agents on any Mac, any contributor creating PRs. No exceptions.

## Site Health Monitoring

Every patrol cycle should also check deployment health:
```bash
curl -s -o /dev/null -w "%{http_code}" {{SITE_URL}}
```
If the site returns non-200 or times out, follow `ops/runbooks/incident-response.md`.
